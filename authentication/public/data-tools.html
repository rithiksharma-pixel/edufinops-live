Export / Import data

Bulk data in and out — for migrations, backups, or offline analysis.

Export

Import leads

Upsert by phone: student_phone is matched against existing leads — a match updates that lead (a blank cell leaves the existing value untouched, it's never cleared), no match creates a new one. student_phone is the only truly required column — everything else has a sensible default on a new lead: blank student_name uses the phone number as a placeholder, blank source_name uses a fallback "Unknown" source, blank loan_amount_requested is left unset.

Other columns: student_email, course_name, university_name, destination_country. source_name must exactly match an existing lead source's name.

Optional — for a historical migration (a lead that already has real history, not a brand-new lead created today):

  • stage_name — the lead's current pipeline stage, matching an existing stage's name exactly (e.g. "Documents Received", "Sanctioned"). Blank = the opening stage, same as a fresh lead.
  • assigned_rm_name — full name of the Relationship Manager this lead is currently assigned to (must be an existing, active RM; if two active RMs share a name, use assigned_rm_email instead to disambiguate). Blank = unassigned.
  • created_date — the lead's real creation date (YYYY-MM-DD), so it doesn't show up as "created today" in every dashboard/report. Blank = now.
  • loan_type — Collateral or Non Collateral. Blank = unset.
  • consultancy_name / consultancy_other_name — which consultancy referred this lead. consultancy_name is matched leniently against existing consultancies (ignoring case, extra spaces/punctuation, and suffix words like "Consultancy"/"Consultant"/"Pvt Ltd") — e.g. "ABC" will auto-match "ABC Consultancy" if that's the only close candidate; rows fuzzy-matched this way are called out in the preview so you can double-check before importing. Use consultancy_other_name for one not yet in the system at all. One of the two is required when source_name is "BD Partnership".

Every row is validated independently — one bad row is reported and skipped, it never blocks the rest of the file. Download the template below to see the full column set with an example historical row filled in.

Import consultancies

CSV columns: name, is_active. name is matched case-insensitively against existing consultancies — a match updates is_active, anything else creates a new consultancy. is_active is optional (TRUE/FALSE; blank defaults to active on new rows, leaves it unchanged on updates).

Import lenders & branches

Required columns: lender_name, lender_code. Optional: branch_name, contact_email, contact_phone.

  • One row per branch — leave branch_name blank for a lender-only row (no branch created on that row).
  • Multiple rows sharing the same lender_code add multiple branches under one lender.
  • lender_code is the match key for an existing lender (case-insensitive) — if it already exists with a different lender_name, that row is rejected rather than silently renaming it.
  • contact_email / contact_phone are lender-level — set them on any one row for that lender_code.

Import deal history

One row per lead + lender. Required: student_phone (must match an existing lead — import leads first), lender_name (must match an existing lender), current_stage_name (Bank Prospect / Login / Sanction / PF / Disbursement / Closed Won).

Upsert by student + lender: a match against an existing deal updates it in place; no match creates a new deal (same as clicking Share in the Lenders tab) and advances it straight to current_stage_name.

  • branch_name — the lender's region/branch this deal is under (optional, must match an existing branch for that lender).
  • current_disposition_name — the disposition within current_stage_name (optional, must match an existing disposition for that stage).
  • Bank Prospect: region_shared_date, sm_shared_date, rm_shared_date, eligibility_status.
  • Login: loan_required_amount, login_amount, login_date, probable_sanction_date.
  • Sanction: sanction_amount, sanction_date, probable_pf_date, interest_rate, tenure_months, moratorium_months.
  • PF: pf_amount, pf_date, probable_disbursement_date.
  • All stage-detail columns are independent of current_stage_name — a deal now at PF can still carry its earlier login_date/sanction_date for a complete history. Blank cells are left unset, never cleared on an update.
  • Disbursement (optional, one tranche per row): disbursement_amount, disbursement_date together, plus optional disbursement_tranche_number (default 1) and academic_term. A deal with multiple historical tranches needs a second import pass — this only records one per row.
  • Hold / Reject (optional overlays): is_on_hold + hold_reason_name (required together), is_rejected + rejection_reason_name (required together).
  • This import does not backdate a full stage-by-stage timeline — only the current stage/status and the detail dates you give are recorded, not synthetic history for stages the deal already passed through.

Bulk update users

CSV columns: email, role_name, reporting_manager_email, team_name, is_active. email identifies the existing user to update — everything else is optional; leave a cell blank to leave that field unchanged. Put NONE in reporting_manager_email or team_name to clear it. is_active must be TRUE or FALSE if set. This only updates existing users — invite new people from Manage Users.
